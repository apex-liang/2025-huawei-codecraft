## 题目概述
关于题目的详细内容请看项目附件，赛题每个阶段的题目都有所不同，但是最基础的部分是相同的，这里做出简要叙述。
本赛题抽象了一个磁盘系统，磁盘系统有N个硬盘，每个硬盘的大小为V。 在每个时间片内（单位时间），会有时间同步标志、删除、写入、读取请求，输入为文本，输出也为文本。

1. 首先会给出磁盘大小、磁盘个数、token个数、标签个数等基本信息

2. 且将所有时间每1800个时间片为一个时间段。接下来的`3*NLABEL`行会依次给出，label1 label2 label3...的每个时间段的删除操作数量（这是NLABEL行）、label1 label2 label3...的每个时间段的写入操作数量（这是NLABEL行）、label1 label2 label3...的每个时间段的读取操作数量（这是NLABEL行）。到这里，选手需要输出OK。

3. 每个时间片的格式为`TIMESTAMP current_timestamp`、`n_delete+delete的对象编号，每行一个编号`、`n_write+写入对象的编号、大小、label，每行一个对象`、`n_read+请求编号、请求读取的对象的编号`,对于时间同步标志、删除、写入、读取请求要求输出格式为`TIMESTAMP current_timestamp`、`被取消的读请求的数量+被取消的读请求的id（这个id一定是刚被删除的对象的读请求id）`、`输出包含4 ∗ 𝑛_𝑤𝑟𝑖𝑡𝑒行，每4行代表一个对象：第一行是对象编号、第二三四行分别是副本123的存储磁盘的磁盘位置`、`前N行为每个磁盘的磁头动作、后面有n_rsp+n_rsp行表示完成的读请求数量和读请求id`


## 策略亮点解读

整个项目的最终使用的核心策略集中在2025-code-craft-semifinals-0402/demos/cpp/v2文件夹中。决赛只针对于赛题做出了适应性改变并无太多调优。

### 存储策略

1. **多级竞争存储策略**：每个磁盘的备份一存储区域都被切成了固定大小的切片。对象块写入的时候优先去自己已经抢占的(disk,block)中去写，其次抢占空闲块以扩容，最后抢占已经被占用的切片。这种机制既保证了同标签数据的物理聚合、降低了碎片化，同时也保证了存储一定存的下。
2. **读取热度对齐** ：根据“性价比”（读取压力与存储量之比）进行静态指引，将时间维度上最同步的 Label 强制布局在相邻物理扇区。从而在读取的时候提高磁盘内读取密度，从而**减少磁针跳转**。
3. **磁盘间负载均衡**：由于每个Label被预分配时为跨盘轮询分配，分配空间来自多个disk存储，所以兼顾了**磁盘间负载均衡**。

### 磁针调度策略

1. **确定性读取路径**：在每个时间段起始，确定第一副本（主副本）作为主读区域，并选择读取“性价比”最大的标签，最大程度压缩磁针的寻道半径。
2. **双磁针动态负载均衡**：引入**滞后反馈机制**，监控双针实时请求处理比率，仅当偏差超过 某设定阈值时触发边界平移，从而动态调整磁针读取负载，平衡了负载收益与 `Jump` 指令的令牌损耗。
3. **全局位图导航**：借鉴 OS 磁盘管理，构建 `read_map` 前缀和索引。在每一个时间步长，磁针通过 O(1)时间复杂度的区间探测，自主决定执行 `Read`（覆盖读取）、`Pass`（小跨度滑行）还是 `Jump`（长距离跳转），获得最优的磁针动作。

### 过载保护

​		算法具备“代价感知”能力。在读取前预判磁针当前位置到目标的寻道时间，若判定在105 个时间片（TTL）内无法完成读取，则主动标记为 `Busy`。

**关于v3版本的详细说明**

1. **热度匹配计算**

     - **动态存量计算**：`demand[i][j]`计算的是第 i 个标签在 j 时刻的存量，公式为：
     
       $$
       Current = (Write + Previous\_Demand) - Deleted
       $$
     
     - 	**峰值识别**：记录每个标签的需求峰值（`max_demand`）以及达到峰值的时间点（`label_top_time`）。
     
     - 	**标签关联热度**：`hot_allocate` 函数通过计算不同标签之间读取需求的“协方差”（`read_demand[i] * read_demand[j]`），评估哪些标签最不容易同时读取，从而在分配存储扇区的时候做负载均衡。
     
2. **磁盘分区与预分配**

   - 每个磁盘分为四个区域，Label A的备份一区域、Label B的备份一区域、LabelC与LabelD备份二混存区域，其他未分配的空闲区域
   - Label A采取最大存量需求排序的方式，选出需求最多的，划分[1,size LableA/2]或者[1,size Lable A]大小的给第一块
   - Label B则根据标签关联热度，选出与当前disk存储的Lable A读取热度冲突最小的，用于负载均衡
   - Label C与Label D选取与Label A、Label B读取热度最不重合、最无关的，用于负载均衡，大小分配按照size C+size D的最大需求乘某系数计算，目的是防止某磁盘的Label A与B在某时刻请求非常少从而浪费了该磁盘
   - 为有效利用备份三，由于之前预分配必然造成在Label A或者Label B的存储密度下降的时候，出现一些碎片化的空间，所以选取相近热度的Label E、F等，存进Label A的区域，使得尽可能在Label A热度下降但仍然很热的时候，Label E、F可以顺便被读取到

3. **负载均衡说明**

   负载均衡的作用是让磁盘的读取负载更为均衡，所以主要依赖于处于同一读取高峰的数据被分开存储，所以Label A与Label B的读取高峰是错开的，Label AB与Label CD的读取高峰是错开的。

4. **磁盘写入**

   磁盘写入主要按照之前**磁盘分区与预分配**所述分配到的区域存储。

5. **磁盘读取**

     在磁盘预处理和写入的过程中，已经尽量把数据备份集中写入到磁盘的区域1、2、3的位置，所以对于每个时间段要有所舍弃的读取。

     定义了一个“性价比”函数，衡量读取某Label的单位小块平均可以完成的request数量

     $$
     xjb = \frac{Read\_Demand}{Max\_Size}
     $$

     计算每个区域的性价比，定义每个磁盘在每个时间段的磁针寻道区间。在每个时间段，磁针只在相应寻道区间内运动。每次运动按照公式计算是否r(这种密度下，基本上连续r是最划算的)

6. **删除操作**

   删除操作只需要维护相应的变量即可

### 预赛
预赛与训练赛题目一样，只是换了数据。预赛相比于训练赛修改了数据的写入峰值分布，由于每个label的存储空间都按照所有时间段的最大需求进行分配必定会存在空间申请超量的问题，典型空间换时间，但新数据的max_demand都很大就会使得原本的静态存储容易存不下而炸盘，如果妥协允许部分乱序存放则会导致磁盘存放碎片化严重，导致处理读请求的速率减慢。所以预赛的主要优化集中在，怎样才能存的下，如何存放数据并使得碎片化并不严重，同时考虑磁盘间负载均衡。

|版本|优化点|跑分|时长|
|---|---|---|----|
|v3|如前文|0(存不下)|/|
|v4|由于v3会炸盘所以做出一定量修改，在v3基础上，让Label A和Label B备份一存储在一个区域里，存储大小需求为(A+B)最大需求，A和B分别是从前往后写和从后往前写，从而尽量聚合写入。备份二预划定区域为Label C和Label D最大需求量的一半。|7496798.6675 (54.1378%)|26.9060s|
|v5|静态的存储容易炸盘，不够灵活，于是想到动态竞争与抢占分配。这里只存储了备份一，从而保证寻道长度最小。每个磁盘的备份一存储区域都被切成了固定大小的切片，这些切片初始状况下不属于任何label。写入的时候优先去自己抢到的(disk,block)中去写，其次抢占空片，其次抢占别人已经占到的片。读取则优化为每个时间段读取热度很大的label，不需要读取的直接跳过，需要读取的分为稠密存储全部r和稀疏存储rp结合的方式。（这个方案去掉了热度匹配，负载均衡方面有缺陷，但总体来说，磁盘存储碎片化大幅度降低，而且因为抢占出现了软均衡（比如抢占空白扇区一般来说都是分散在不同磁盘的），于是有分数提升|9563278.7375 (69.0608%)|23.3770s.|
|v6|相比于v5在负载均衡方面做出改进，预分配相应大小的block，每次轮询使得负载均衡|9626336.7850 (69.5162%)|26.7270s|

### 复赛

复赛相比预赛多了垃圾回收机制和罚时机制，以及增加了双磁针。双磁针对读取非常重要，以及在合适的时间busy某些请求以减少罚时也非常重要。
|版本|优化点|跑分|时长|
|---|---|---|----|
|v1|基于前述v6稍作改进，多加了一个磁针，两个磁针各自读备份一的一半区域。超时处理采用不打算读的label的request直接Busy避免罚时，以及超时busy两部分。未利用交换机制。|18802016.2424 (58.2822%)(read:61.4222%, busy:-3.1400%)  |58.6140s|
|v2|预分配使用热度匹配，使得同时到达高峰读取的label可以被存储到磁盘中相近的位置，从而减少磁针跳转。针对“双磁针”设计了动态负载均衡策略，程序记录了两个磁针分别完成的请求数 `req_p1` 和 `req_p2`，由于磁针分界线的移动会导致jump所以谨慎对待，当req比值大于1.2的时候才会移动 `disk_point_mid` ，移动距离与请求数量差距有关。gc整理每个块内，将块内数据紧密存放没有空隙。|23033871.3773 (71.4000%)(read:71.5033%, busy:-0.1032%)|53.1640s|

v2核心优化说明

1. **磁盘间负载均衡**：热度分配与竞争并存的存储策略，从而使得负载更集中以减少跳转/寻道长度
   1. 竞争存储策略使得存储一定存的下且碎片化大大减小
   2. 根据“性价比”的预分配使得相同读取峰值的label得以被存在一起，从而在读取的时候，相邻区域的大概率被同时读取，从而减少磁针跳转。
   3. 由于每个Label被分配给了多个disk存储，所以兼顾了负载均衡
2. **磁盘内磁针负载均衡**：针对“双磁针”设计了动态负载均衡策略
   1. 每个时间段的起始先规定磁针1和磁针2的扫描范围
   2. 每隔一小段时间根据两个磁针分别完成的请求数 `req_p1` 和 `req_p2`动态调整磁针扫描范围。由于磁针分界线的移动会导致jump所以谨慎对待，当req比值大于某阈值的时候才会移动`disk_point_mid` 
   3. 按需移动`disk_point_mid` 
   4. 使用类似于操作系统中磁盘的bitmap类似的readmap，用来导航与记录磁盘中需要读取的块，并在read逻辑中确定磁针最佳运动轨迹。
3. **回收机制**：
   1. gc整理每个块内，将块内数据紧密存放没有空隙。

### 总决赛

​	总决赛在前面的基础上增加了第二轮读取，所以第二轮如何应用已知数据是非常重要的。但由于时间不足，所以总决赛的策略有问题，读取效率并不高。这里我只上传了最后的版本，该版本经过后续确认与大多队伍在第一轮读取中没有显著差异，在第二轮中过于落后。practice_1得分46169741.3086 (62.8779%)(read:63.5825%, busy:-0.7046%)用时295.3320s。
